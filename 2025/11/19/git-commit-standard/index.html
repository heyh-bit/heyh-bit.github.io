git_commit_standard

在使用 Git 进行版本控制时，编写规范的提交（commit）信息是团队协作和项目维护的重要环节。一个标准的 commit message 能够清晰地表达修改内容、便于追踪问题、生成 changelog，并支持自动化工具解析。

以下是目前广泛采用的 Commit Message 标准规范，主要基于 Conventional Commits 规范，并结合社区最佳实践：

一、Commit Message 基本格式

1. Header（头部）：必须

  • type：提交类型，表示本次变更的类别。
  • scope（可选）：变更影响的范围或模块（如 auth, router, ui, docs 等）。
  • subject：简洁的变更描述，使用动词开头，小写，不加句号。

🔹常见 type 类型：

type 含义说明
feat 新功能（feature）
fix 修复 bug
docs 文档更新
style 代码格式调整（不影响逻辑，如空格、分号等）
refactor 重构代码（既非新增功能也非修复 bug）
perf 性能优化
test 添加或修改测试
build 构建系统或依赖变更（如 webpack、npm）
ci 持续集成配置修改（如 GitHub Actions, Travis）
chore 其他杂项任务（如更新脚手架配置）
revert 回滚某个 commit
🔹 范围（scope，可选）：指明影响的模块、组件或功能区域，例如：
  • user（用户模块）
  • login（登录页）
  • api（接口层）
  • header（头部组件）
  • router（路由配置）
    🔹 简要描述（subject）
  • 使用动词开头，简洁明确
  • 不超过 50 字
  • 小写开头，结尾无句号
  • 描述“做了什么”，而不是“改了哪些文件”

三、书写建议（Best Practices）

  1. 使用祈使语气（Imperative mood）
    ❌ “Fixed bug” / “Fixes bug”
    ✅ “fix bug”

  2. 首字母小写，结尾无句号
    ✅ update dependencies
    ❌ Update dependencies.

  3. 保持简短但清晰
    主题行建议不超过 50 字符，正文每行不超过 72 字符。

  4. 避免空白或模糊信息
    ❌ update file
    ✅ fix: correct null reference in user service

  5. 关联 issue 或 ticket
    在 footer 中添加 Closes #123 或 Fixes PROJ-456，便于自动关闭问题。

四、工具支持（推荐）

  • commitlint + husky：校验 commit message 是否符合规范。
  • Commitizen：交互式生成标准 commit。
  • VS Code 插件：如 “GitLens” 或 “Better Commit Policy” 辅助编写。

总结

要素 是否必须 建议格式
type ✅ 是 feat, fix, docs 等
scope ❌ 否 模块名，括号内
subject ✅ 是 小写动词开头，无句号
body ❌ 否 解释 why，非 how
footer ❌ 否 BREAKING CHANGE 或 issue 链接

遵循标准的 commit 规范，不仅能提升代码可维护性，还能为后续的自动化发布（如 semantic-release）、生成 changelog 提供极大便利。
